## Summary

Phase 04.1 is a pure visual polish pass — no new pages, no new content, no new components. The goal is to close the gap between the current Tailwind-styled implementation and the original yapu.solutions site at the pixel level. Based on direct analysis of the live site and the current codebase, the gaps fall into five categories: (1) navigation background color is wrong (current: Mint/`bg-accent`, original: White), (2) hero sections are missing their distinctive dark-teal full-bleed treatment on some pages and the correct typography scale, (3) body section spacing and heading styles are too generic, (4) button shapes need pill rounding (`border-radius: 100em` not `rounded-lg`), and (5) the Impact page hero needs a gold gradient unique to that page.

All fixes are Tailwind class changes to existing components and page files. No new npm packages are needed. No structural refactoring is needed. The work is mechanical but spread across 9 files (7 pages + Navigation + Hero components). The biggest risk is the navigation color decision (WHITE background, not Mint) which contradicts the Phase 03 decision log entry that said "Nav background is bg-accent (Mint #45B5B4) — matches live YAPU site." The live site was re-examined for this research and the nav on yapu.solutions is clearly WHITE with dark-teal text — not mint. That Phase 03 decision was incorrect and must be reversed.

**Primary recommendation:** Fix navigation background first (highest visual impact, most obvious mismatch), then apply pill-style buttons site-wide, then correct hero sizing, then section spacing and typography scale. Work page-by-page with a side-by-side browser comparison at 1440px as the guide.

### Pattern 1: Pill-shaped buttons (border-radius: 100em)

**What:** yapu.solutions uses `border-radius: 100em` (pill shape) for ALL CTAs and buttons. Current implementation uses `rounded-lg` (8px radius).

**When to use:** Every `<a>` and `<button>` that is a CTA — hero CTAs, service CTAs, filter buttons, submit buttons, YAPU App nav button.

**Example:**
```tsx
// BEFORE (wrong):
className="bg-cta text-white px-8 py-3 rounded-lg font-semibold ..."

// AFTER (correct):
className="bg-cta text-white px-8 py-3 rounded-full font-semibold ..."
```

Tailwind's `rounded-full` maps to `border-radius: 9999px`, which is functionally identical to `100em` for buttons.

### Pattern 2: Navigation — White background, dark-teal text

**What:** The original yapu.solutions nav bar is WHITE (#ffffff) with dark-teal text/logo. The current implementation uses `bg-accent` (Mint #45B5B4) which is WRONG. The Phase 03 decision log was incorrect — re-examination of the live site confirms white nav.

**Correct nav styling:**
```tsx
// BEFORE (wrong):
className={cn('sticky top-0 z-50 bg-accent text-foreground transition-shadow', ...)}

// AFTER (correct):
className={cn('sticky top-0 z-50 bg-white text-foreground border-b border-border transition-shadow', ...)}
```

Nav items should be dark-teal (`text-brand`) and the YAPU App CTA button keeps its orange-red (`bg-cta`) pill shape.

### Pattern 3: Hero — Dark Teal full-bleed with large typography

**What:** yapu.solutions hero uses `rgb(30,90,100)` = Dark Teal (#1E5A64 = `bg-brand`) as background, white text, large font (2em–3em), bold weight (900 equivalent = `font-black` or `font-extrabold`), and notable letter-spacing (0.08–0.1em). Current hero uses `text-4xl md:text-5xl font-bold` which is correct size but missing letter-spacing and needs the subtitle to be lighter weight.

**Correct hero pattern:**
```tsx
// Hero heading — add tracking and use heavier weight
<h1 className="text-5xl md:text-6xl font-black mb-6 tracking-wide leading-tight">
  {title}
</h1>
// Subtitle — lighter weight (original is weight 300)
<p className="text-lg md:text-xl mb-8 opacity-90 max-w-2xl mx-auto font-light leading-relaxed">
  {subtitle}
</p>
// CTA button — pill shape
<a className="bg-cta text-white px-8 py-3 rounded-full font-semibold hover:opacity-90 transition-opacity inline-block text-lg">
```

### Pattern 4: Impact page — Gold gradient hero variant

**What:** The Impact page on yapu.solutions uses a GOLD/YELLOW gradient for its hero: `background: linear-gradient(90deg, rgba(213,186,0,1) 0%, ...)`. All other pages use the standard dark-teal hero. The Impact hero is a unique design choice specific to that page.

**Implementation approach:** Hero component needs to accept an optional `variant` prop OR the Impact page renders its own hero section directly (simpler). Since Hero is a shared component, use an optional `bgClass` prop override:

```tsx
// Hero.tsx — add optional bgClass prop
interface HeroProps {
  title: string;
  subtitle: string;
  ctaText: string;
  ctaHref: string;
  bgClass?: string; // override background, default = 'bg-brand'
}

// impact/page.tsx — pass gold gradient
<Hero
  title={...}
  subtitle={...}
  ctaText={...}
  ctaHref={...}
  bgClass="bg-[#D5BA00]" // or inline style for gradient
/>
```

For the gradient, use inline style since Tailwind can't express the exact gradient easily:
```tsx
// In Hero.tsx
<section
  className={`text-white py-20 px-6 ${bgClass || 'bg-brand'}`}
  style={bgClass ? undefined : undefined}
>
```

Actually the cleanest approach: accept a `style` prop for background overrides. Or use `data-variant` + CSS. The simplest and most Ember-editable is an inline style prop on the Impact page's hero section rendered directly (skip the Hero component for Impact and write the section inline with `style={{ background: 'linear-gradient(...)' }}`).

### Pattern 5: Section spacing — Increase padding to match original

**What:** Original sections use 2–4em vertical padding. At 1rem base, `py-16` (4rem) and `py-20` (5rem) are appropriate. Current implementation already uses `py-16` consistently, which is close. Increase hero to `py-24` or `py-28` (original is very tall). Internal sections need `max-w-7xl` rather than `max-w-4xl` for wider content areas.

**Container width decisions:**
- Full-width content: `max-w-7xl` (1280px)
- Text-heavy sections: `max-w-4xl` (896px)
- Grid sections: `max-w-6xl` (1152px)

### Pattern 6: Typography — Uppercase section headings with letter-spacing

**What:** Original uses `text-transform: uppercase` with `letter-spacing: 0.1em` for certain section titles and category labels. Not all headings — mainly labels/eyebrows and some prominent section titles. Body text weight is 300 (light).

**Apply to:** Category labels in service cards, section subtitle labels, filter button text.
```tsx
// Uppercase label treatment
<span className="text-xs font-semibold uppercase tracking-widest text-muted-foreground">
  {label}
</span>
```

### Pattern 7: Section alternation — Light gray vs. white (not muted/30)

**What:** Original uses `rgb(248,248,248)` (very light gray) and `rgb(255,255,255)` (white) for alternating sections. The current implementation uses `bg-muted/30` (30% opacity muted) which renders differently. Use `bg-gray-50` or `bg-neutral-50` (same as `#f9fafb`) as the closest Tailwind v4 equivalent.

```tsx
// BEFORE: bg-muted/30
// AFTER: bg-gray-50 (light sections) or bg-white/bg-background (white sections)
```

### Pattern 8: News filter buttons — Teal pill buttons

**What:** News category filter buttons on original use `rgb(69,181,180)` (Mint) background with pill shape (`border-radius: 100em`). Active state is white bg with teal text. Current implementation needs to adopt pill shape and match color.

### Anti-Patterns to Avoid

- **`rounded-lg` on CTAs:** All CTA buttons must be `rounded-full` (pill), not `rounded-lg`
- **`bg-muted/30` for alternating sections:** Visually different from the original's `#f8f8f8`. Use `bg-gray-50` instead
- **`bg-accent` on nav:** Nav must be white. Mint should only appear as accent on interactive elements
- **Hardcoded hex in style attributes for brand colors:** Use CSS custom properties or Tailwind tokens for brand colors; only use inline style for the Impact hero gradient which has no token equivalent

## Common Pitfalls

### Pitfall 1: Nav Color Contradiction with Phase 03 Decision

**What goes wrong:** Phase 03 STATE.md records "Nav background is bg-accent (Mint #45B5B4) — matches live YAPU site." But this is wrong. The live yapu.solutions nav is white with dark-teal text. Changing it to white is a direct contradiction of a logged decision.

**Why it happens:** The Phase 03 decision was made without careful examination. Mint IS used in the nav as an accent (hover states, active states), but the base background is white.

**How to avoid:** Treat the live site at yapu.solutions as the ground truth, overriding any prior decision that contradicts it. Document the correction in STATE.md.

**Warning signs:** If nav looks like a teal/mint banner instead of a clean white header, it's wrong.

### Pitfall 2: Forgetting the Impact page's unique gold hero

**What goes wrong:** Developer applies the standard dark-teal `bg-brand` hero to Impact page, missing its gold gradient.

**Why it happens:** The Hero component defaults to `bg-brand`. Impact page just passes props without a background override.

**How to avoid:** Impact page must explicitly override the hero background. Add a `bgClass` prop (or `style` prop) to the Hero component.

**Warning signs:** Impact page hero is dark teal instead of gold/yellow.

### Pitfall 3: Over-polishing vs. "close enough"

**What goes wrong:** Spending too long pixel-matching font sizes, padding values, and line-heights that are imperceptible to the human eye.

**Why it happens:** "Pixel-accurate" sounds like it means measuring every value.

**How to avoid:** The standard is "visually equivalent at 1440px and 375px" — meaning a human looking at both cannot immediately spot the difference. Near-identical is sufficient. Don't spend more than 15 minutes on any single element.

### Pitfall 4: Breaking mobile layout while fixing desktop

**What goes wrong:** Adding larger font sizes or padding for desktop breaks the 375px mobile layout.

**Why it happens:** Using fixed sizes instead of responsive Tailwind prefixes.

**How to avoid:** All size changes must use responsive prefixes: `text-4xl md:text-5xl lg:text-6xl`, `py-16 md:py-20 lg:py-24`. Test at both 375px and 1440px after every change.

### Pitfall 5: The SVG logo visibility issue on white nav

**What goes wrong:** YAPU SVG logo has a white `.st0` fill, which makes it invisible on a white nav background.

**Why it happens:** The logo was designed for dark/colored nav backgrounds.

**How to avoid:** On white nav, the logo needs either: (a) a dark version of the SVG, or (b) a color inversion/filter applied. Check whether a dark-fill variant exists in `public/logos/`. If not, apply CSS filter: `filter: brightness(0)` makes white SVG black. Or use `invert` on the logo for white-on-teal nav... wait — reversing to white nav means we need the logo to be dark. Apply `filter: invert(1)` or `filter: brightness(0)` to logo on white background.

**Warning signs:** Navigation renders with no visible logo on white background.

## Open Questions

1. **Navigation logo on white background**
   - What we know: SVG has white fill; currently visible on mint nav but invisible on white nav
   - What's unclear: Is there a dark-logo variant SVG in the project or on the original site?
   - Recommendation: Apply `filter: brightness(0) saturate(100%)` CSS to force dark rendering. Check `public/logos/` first for a dark variant.

2. **Service module card visual treatment**
   - What we know: Original site cards are relatively minimal with icons
   - What's unclear: Exact card border/shadow treatment — the WebFetch shows "light gray backgrounds with subtle drop shadows"
   - Recommendation: Keep current shadcn `Card` component. Add `hover:shadow-lg` and possibly mint top border `border-t-4 border-accent` to distinguish cards.

3. **Font approximation ceiling**
   - What we know: Original uses "museo-local" (proprietary), current uses Nunito (Google Fonts). VIS-05 marks system fonts as "complete" — this was accepted at Phase 1
   - What's unclear: How close is Nunito to the original at 1440px? Museo Sans Rounded is slightly rounder/tighter
   - Recommendation: Apply higher `font-black` weight (900) + `tracking-wide` on headings to close the gap further. No new fonts.

4. **Section padding precision**
   - What we know: Original uses 2–4em vertical padding (~32–64px). Current uses `py-16` (64px = 4rem)
   - What's unclear: Some sections may need `py-20` or `py-24` to match taller sections
   - Recommendation: Normalize to `py-16 md:py-20` for standard sections, `py-20 md:py-28` for hero.
